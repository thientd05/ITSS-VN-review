1 Bốn cách để code "đúng"

Verification / QA

Debugging

Defensive programming

Lập trình với tinh thần validation & debugging sẵn trong đầu.

Test ≠ Debug

Test: tiết lộ sự tồn tại vấn đề (+tăng độ tin cậy). Debug: xác định vị trí + nguyên nhân.

Trích dẫn"Any fool can write code that a computer can understand. Good programmers write code that humans can understand." — Martin Fowler.

2 Programming Style (coding convention)

  • Dùng hằng số cho mọi giá trị hằng (VD thuế); khai báo hằng trước biến ở đầu thủ tục.
  • Dùng biến khi có thể; đặt tên mô tả (có/không prefix, VD btnDone).
  • Comment code (nhất là chỗ mơ hồ/dễ hiểu nhầm) nhưng không lạm dụng.
  • Dùng ngoặc cho biểu thức toán học dù không bắt buộc: (3*2) + (4*2) → rõ ý.
  • Viết code hiệu quả; form input/output thân thiện; kèm header đầu file (tên tác giả, ngày, project, lớp, mô tả…).
Công cụJava Coding Conventions; công cụ Checkstyle giúp tuân chuẩn.

3 Code Tuning / Optimization

Code tuningSửa code đúng để chạy hiệu quả hơn. KHÔNG phải cách rẻ/hiệu quả nhất để cải thiện performance. Quy tắc 80/20: 20% method tiêu tốn 80% thời gian chạy.

Myths (đều SAI!)

  • "Giảm số dòng code high-level thì code máy nhanh/nhỏ hơn" — sai.
  • "Chương trình nhanh cũng quan trọng như chương trình đúng" — sai.
  • "Một số phép toán chắc chắn nhanh/nhỏ hơn cái khác" — sai → phải luôn đo!
  • "Nên tối ưu ngay khi viết (as you go)" — sai (khó tìm bottleneck khi chưa chạy xong; tối ưu sớm làm xao nhãng mục tiêu khác).
Khi nào tuneDùng thiết kế chất lượng cao trước: (1) Make it right · (2) modular & dễ sửa · (3) khi hoàn chỉnh & đúng mới kiểm performance. Cân nhắc compiler optimization; Measure; viết code sạch dễ hiểu & sửa.

Measurement cần: chính xác, lặp lại được. Optimize theo vòng lặp: đo cải thiện sau mỗi lần tối ưu; nếu không cải thiện → revert.

Kỹ thuật tuning

  • Dừng test khi đã biết câu trả lời (short-circuit).
  • Sắp test theo tần suất (order tests by frequency).
  • Unswitching loops (đưa if ra ngoài vòng lặp).
  • Giảm việc bên trong vòng lặp (cache rates->discounts->factors->net).
  • Khởi tạo ở compile-time (hằng số).
  • Lazy evaluation (tính khi cần).
  • Khi duyệt dữ liệu, giữ tham chiếu bộ nhớ tuần tự (sequential).

4 Refactoring

Refactoring — Martin Fowler"Cải thiện thiết kế & chất lượng của mã nguồn hiện có mà không thay đổi hành vi bên ngoài." Là quy trình từng bước biến bad code thành good code, dựa trên refactoring patterns.
Khi nào refactor?Bad smells báo hiệu cần refactor: để thêm chức năng dễ hơn · khi fix bug · khi review code người khác · khi có technical debt · khi làm TDD. Unit test bảo đảm refactoring không đổi hành vi — nếu chưa có test thì viết test trước.

Nguyên tắc & quy trình

Main PrinciplesKISS (keep it simple) · DRY (avoid duplication) · self-documenting · giảm tổng lượng code · tách concern (decoupling) · mức trừu tượng phù hợp · Boy scout rule (để code sạch hơn lúc bạn tới).
Typical Process(1) Lưu/backup code hiện tại · (2) Chuẩn bị test bảo đảm hành vi · (3) refactor từng cái một, giữ nhỏ · (4) chạy test, pass thì tiếp/không thì revert · (5) check-in.

5 Code Smells — 6 nhóm

Code smellCác cấu trúc trong code gợi ý khả năng cần refactoring.
NhómCác smell tiêu biểu
Bloaters (phình to)Long method · Large class · Primitive obsession · Long parameter list · Data clumps · Combinatorial explosion · Oddball solution · Class doesn't do much · Required setup/teardown code
Obfuscators (che khuất)Regions · Comments (nên nói WHY chứ không WHAT/HOW) · Poor names · Vertical separation · Inconsistency (POLA) · Obscured intent
OO AbusersSwitch statement (→ polymorphism) · Temporary field · Class depends on subclass (circular, phá Liskov) · Inappropriate static field
Change PreventersDivergent change (phá SRP) · Shotgun surgery · Conditional complexity (deep nesting) · Poorly written tests
Dispensables (thừa)Lazy class · Data class · Duplicated code (phá DRY) · Dead code · Speculative generality (YAGNI)
CouplersFeature envy · Inappropriate intimacy · Law of Demeter (LoD) · Indecent exposure · Message chains · Middle man · Tramp data · Artificial coupling · Hidden temporal coupling · Hidden dependencies
Refactoring theo cấp độData-level: replace magic number bằng constant, rename biến, convert set of type codes → enum. Statement-level: decompose boolean, dùng break/return, replace conditional bằng polymorphism, null-object pattern. Method-level: extract/inline method, rename, add/remove parameter. Class-level: pull up/push down, extract subclass, replace inheritance ↔ delegation. Class Interface: extract interface, encapsulate exposed field, remove middle man. System-level: replace error codes bằng exception, dependency injection, apply architecture patterns.

6 Debugging

Thuật ngữ: Error → Bug → Fault → Defect → Failure. "Bug" đầu tiên thực sự là một con bướm đêm kẹt trong relay (Grace Hopper).

6.1 Defense in depth (phòng thủ nhiều tầng)

TầngÝ tưởng
1. Make errors impossibleTrong ngôn ngữ (Java chặn memory overwrite; BigInteger không tràn), protocol/library (TCP/IP không đảo thứ tự), quy ước tự đặt (cấm đệ quy, immutable data). "Impossible by design".
2. Don't introduce defects (Correctness)Làm đúng ngay từ đầu; nghĩ trước khi code; Simplicity là chìa khóa (modularity, ADT interface rõ, defensive programming, viết spec cho mọi module).
3. Make errors immediately visibleFail-fast: assertions bắt lỗi sớm; unit test cô lập module; regression test. Khi lỗi được khoanh vào một method nhỏ → đọc code là tìm ra.
4. Debugging (last resort)Cần khi failure (hậu quả) xa cause (defect). Dùng phương pháp khoa học.
Fail-fast — Đừng che giấu lỗiĐừng làm code "robust" giả tạo bằng cách không fail. Assertion giúp document & kiểm invariant: biến lỗi thành failure ngay. Lỗi phát hiện càng sớm càng dễ sửa (giảm khoảng cách defect ↔ detection). Câu hỏi trong production: có nên giữ assertion? — tùy ngữ cảnh (Ariane 5 crash vì exception overflow không xử lý). Trích: Hoare — "thiết kế đơn giản để hiển nhiên không có lỗi vs phức tạp đến mức không có lỗi hiển nhiên"; Kernighan — "debug khó gấp đôi viết code, nên đừng viết code khôn ranh hết mức".

6.2 Quy trình Debugging (4 bước)

  1. Clarify symptom: tìm test nhỏ, lặp lại được, tái hiện failure (đơn giản hóa input).
  2. Find & understand cause: thu hẹp vị trí; study data → hypothesize → experiment → repeat. Không sang bước sau khi chưa hiểu nguyên nhân.
  3. Fix: typo hay lỗi thiết kế? có xuất hiện ở chỗ khác?
  4. Re-run all tests (thêm test vào regression suite).
Kỹ thuật & scientific methodDebug là quá trình khoa học có hệ thống: formulate hypothesis → design experiment → perform → adjust. Binary search trên code lỗi (kiểm kết quả trung gian ở nửa đường → về defect trong O(log n)). Reducing input size (divide-and-conquer, "very very happy" → "aab"…). Localizing: bỏ dần từng phần tới khi hết failure / thêm dần tới khi failure xuất hiện.
Heisenbugs & hard bugsHeisenbug: chương trình tuần tự deterministic thì failure lặp lại được, nhưng thực tế có input liên tục/timing/concurrency → failure ngẫu nhiên, khó tái hiện (dùng debugger/assertion thì bug biến mất!). Mẹo: rebuild từ đầu; giải thích cho bạn bè; minimize input; add checks; dùng Logging Events (circular buffer) để tái dựng quá khứ. Where is the bug? — tìm lỗi ngớ ngẩn trước (đảo thứ tự tham số Collections.copy(src,dest), @Override bắt typo, a==b vs a.equals(b), recompile everything).
Công cụ (Eclipse plug-in)Checkstyle (tuân coding standard) · FindBugs (static analysis tìm bug & performance issue, >200 rule: Correctness, Bad practice, Performance, Dodgy; input là file .class, dùng "bug patterns").

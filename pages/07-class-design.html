1 Tạo Design Class ban đầu

6 bước của Class Design: Create Initial Design Classes → Define Operations/Methods → Define Relationships → Define States → Define Attributes → Class Diagram. Analysis class → Design element theo ánh xạ many-to-many.

Bao nhiêu lớp là đủ?
Nhiều lớp đơn giản
  • Mỗi lớp đóng gói ít "trí tuệ" hệ thống
  • Tái sử dụng cao hơn
  • Dễ cài đặt hơn
Ít lớp phức tạp
  • Mỗi lớp gánh phần lớn "trí tuệ"
  • Ít tái sử dụng
  • Khó cài đặt hơn
Một lớp nên có MỘT mục đích rõ ràng — "do one thing and do it well".

Boundary class

UI: xét công cụ phát triển UI dùng gì, dựng được bao nhiêu bằng công cụ. External system interface: thường mô hình như subsystem.

Entity class

Object thường passive & persistent; yêu cầu performance có thể buộc re-factoring (VD tách FatClass thành DataHelper/LazyDataHelper).

Control class

Có thực sự cần? Có nên tách? Quyết định theo: complexity, change probability, distribution & performance, transaction management.

2 Định nghĩa Operation & Method

Operation vs MethodOperation: hành vi (từ message trong interaction diagram). Method: mô tả cách cài đặt operation (thuật toán đặc biệt, dùng object/operation nào, cách hiện thực attribute & relationship).

Đặt tên & signature

  • Tên: chỉ rõ kết quả, theo góc nhìn client, nhất quán giữa các lớp.
  • Signature: operationName([direction] parameter: class, ...): returnType. Direction = in (mặc định) / out / inout.
  • Kèm mô tả ngắn, nêu ý nghĩa mọi tham số → program documentation (JavaDoc: @param, @return, @throws).
Guideline signatureXét tham số: truyền by value/by reference, có bị đổi, optional, giá trị mặc định, khoảng hợp lệ. Càng ít tham số càng tốt. Truyền object thay vì "data bits".

Visibility & Scope

Visibility (bao đóng)+ Public · # Protected · - Private. Dùng để cưỡng chế encapsulation.
ScopeInstance scope: 1 bản cho mỗi instance. Classifier scope: 1 bản cho toàn bộ instance (như static) — biểu diễn bằng gạch chân tên attribute/operation.

3 Quan hệ giữa các lớp

Quan hệKý hiệuÝ nghĩa
Associationđường thẳng (có mũi tên navigability)Quan hệ cấu trúc (structural): object của lớp này nối object lớp kia. "use-a".
Aggregationhình thoi rỗng ◇Association đặc biệt whole-part; "has-a / is-a-part". Instance lớp này gồm instance lớp kia.
Compositionhình thoi đặc ◆Aggregation mạnh: part không chia sẻ, chết cùng whole (share life-time).
Generalization (Inheritance)mũi tên tam giác rỗng △"is a kind of". Subclass kế thừa cấu trúc/hành vi của superclass.
Dependencyđường nét đứt có mũi tên ⇢Quan hệ phi cấu trúc, tạm thời.
Từ link → associationMỗi link trong communication diagram → một association trong class diagram ("relationship for every link"). Instance của association là một link.

3.1 Role & Multiplicity

Role = "khuôn mặt" lớp đóng trong association (VD Instructor, Department Head, Prerequisites).

MultiplicitySố instance của lớp này ứng với một instance của lớp kia. Trả lời 2 câu: association bắt buộc hay tùy chọn? Số instance tối thiểu & tối đa liên kết được?
Ký hiệuÝ nghĩa
(trống)Unspecified
1Exactly One
0..* hoặc *Zero or More
1..*One or More
0..1Zero or One (optional)
2..4Specified Range
2, 4..6Multiple, Disjoint Ranges
Cài đặt JavaMultiplicity "many" → dùng Collection: private List<InsuranceContract> contracts;. Chiều 1 → field đơn: private InsuranceCompany refers_to;.

3.2 Association vs Aggregation vs Composition

Khi nào dùng gìHai object gắn chặt whole-part → Aggregation. Thường độc lập dù hay đi cùng → Association. Khi nghi ngờ, dùng Association.
Aggregation (Java)
class Car {
  private List<Door> doors;
  Car(String name, List<Door> doors){
    this.doors = doors; }
  public List<Door> getDoors(){ return doors; }
}
Part (Door) truyền vào từ ngoài — có thể tồn tại độc lập.
Composition (Java)
final class Car {
  private final Engine engine; // composition
  Car(Engine engine){ this.engine = engine; }
  public void move(){ engine.work(); ... }
}
Engine tạo & chết cùng Car — final, sở hữu chặt.

3.3 Navigability

Cho biết có thể "đi" từ lớp nguồn tới lớp đích qua association hay không (mũi tên).

Chọn chiều nào cần thật sự?Khám phá interaction diagram. Ngay cả khi cả 2 chiều "có vẻ" cần, thường một chiều dùng ít / số instance nhỏ → chỉ giữ chiều đó. Chỉ giữ cả hai chiều khi số instance của cả hai lớp đều lớn & cần đi cả hai chiều. Mục tiêu: quan hệ càng "nhẹ" càng tốt.

3.4 Dependency vs Association

AssociationDependency
Bản chấtCấu trúc, lâu dàiPhi cấu trúc, tạm thời (limited duration, context-independent, summary)
Visibility làm cho "biết nhau"Field referenceLocal variable · Parameter · Global reference
3 kiểu visibility của DependencyLocal variable: op1() chứa biến cục bộ kiểu ClassB. Parameter: ClassB truyền vào op1(aParam: ClassB). Global: ClassUtility toàn cục. → Cân nhắc: quan hệ lâu dài = association (field); nhiều object dùng chung 1 instance = parameter/global; không dùng chung = local. Tạo/hủy đắt? → dùng field/parameter/global. "Strive for the lightest relationships possible."

3.5 Generalization

Quan hệ trong đó một lớp chia sẻ cấu trúc/hành vi cho ≥1 lớp; định nghĩa hệ thứ bậc trừu tượng; subclass kế thừa superclass; "is a kind of". Single inheritance / Multiple inheritance.

4 Define States — State Machine

State MachineĐồ thị có hướng gồm state (node) nối bởi transition (cung); mô tả lịch sử đời của một reactive object. Transition: Event(args)[guard condition]/activity.

State

Trạng thái của object. VD CourseInfo: Opened (numStudents ≥ 3) / Closed (< 3).

Event

Từ operation trong interface. VD addLecturer, removeLecturer.

Transition

Sự chuyển state khi nhận event (có thể kèm guard).

Activity (Action)

Entry (khi vào state) · Do (đang chạy) · Exit (khi ra state).

Pseudo-statesInitial state (khi object được tạo, bắt buộc, chỉ 1). Choice (đánh giá động guard sau, chỉ segment đầu có trigger). Final state (kết thúc đời object, tùy chọn, có thể nhiều).
Object nào cần State Machine?Object có vai trò được làm rõ bởi state transitions; use case phức tạp bị điều khiển bởi state. KHÔNG cần mô hình cho: object ánh xạ thẳng sang cài đặt; object không bị điều khiển bởi state; object chỉ có một state tính toán.
Ánh xạ về mô hìnhEvent → operation; method cập nhật thông tin đặc thù state; state thường biểu diễn bằng attribute (đầu vào cho bước Define Attributes).

5 Define Attributes

AttributeattributeName : Type = Default (kèm visibility +/#/-). Type phải là kiểu dữ liệu sơ cấp (elementary) của ngôn ngữ cài đặt (built-in, user-defined type, hoặc user-defined class).

Cách tìm attribute

  • Properties/characteristics của các lớp đã xác định; thông tin lớp cần giữ.
  • "Danh từ" không trở thành lớp: giá trị của nó là điều quan trọng · thông tin thuộc sở hữu duy nhất của object · không có hành vi.
  • Xem mô tả method, xem states, xem thông tin lớp phải tự duy trì.
Derived Attribute (thuộc tính dẫn xuất)Attribute mà giá trị được tính từ attribute khác. Dùng khi: không đủ thời gian tính lại mỗi lần cần, hoặc cần đánh đổi runtime performance với bộ nhớ.

6 Class Diagram & Review

Class diagram = static view. 3 cách dùng khi mô hình static: mô hình vocabulary của hệ thống · collaborations · logical database schema.

Review points
Classes: tên rõ; một abstraction; attribute/behavior gắn kết chức năng; có generalization; đáp ứng yêu cầu; nhất quán với state machine; mô tả đủ vòng đời instance; đủ hành vi.
Operations: dễ hiểu; state đúng; đủ hành vi; tham số đúng; message được gán operation đầy đủ; spec cài đặt đúng; signature theo chuẩn.
Attributes: một khái niệm, tên mô tả, đủ cho Use-Case Realization. Relationships: tên role mô tả, multiplicity đúng.

1 Design Pattern là gì?

Định nghĩa (Christopher Alexander)"Mỗi pattern mô tả một vấn đề lặp đi lặp lại trong môi trường, rồi mô tả cốt lõi của giải pháp — để bạn có thể dùng lại giải pháp đó hàng triệu lần mà không bao giờ làm giống hệt nhau."

Design pattern = giải pháp chuẩn cho vấn đề lập trình thường gặp · kỹ thuật làm code linh hoạt hơn (giảm coupling) · cách nói tắt để mô tả thiết kế (static structure & dynamic behaviour).

GoF — Gang of FourGamma, Helm, Johnson, Vlissides (xuất bản 1994). Empiricist, không lý thuyết suông; quy tắc: không gì thành pattern trừ khi tìm được ≥3 ví dụ thực tế.

3 nhóm pattern GoF

NhómTrừu tượng hóaVí dụ

Cách mô tả một patternElements chính: Pattern Name · Problem (context + điều kiện áp dụng) · Solution (mô tả trừu tượng, các element/quan hệ/trách nhiệm, như template, language-neutral) · Consequences (kết quả & trade-off, ảnh hưởng flexibility/extensibility/portability). Mẫu đầy đủ GoF: Name & classification, Also Known As, Motivation, Applicability, Structure, Participants, Collaboration, Consequences, Implementation, Sample Code, Known Uses, Related Patterns.

2 Singleton (Creational)

SingletonĐảm bảo một lớp có chính xác một instance & cung cấp điểm truy cập toàn cục tới nó.

Cách hiện thực

  • Constructor(s) để private — client không thể new từ ngoài.
  • Khai báo một instance private static của lớp.
  • Cung cấp method public static getInstance() trả về instance duy nhất (có thể cần synchronized để chạy đúng đa luồng).
Eager (khởi tạo sớm)
public class Singleton {
  private static final Singleton
     instance = new Singleton();
  private Singleton(){}
  public static Singleton getInstance(){
    return instance;
  }
}
Lazy (khởi tạo muộn)
public class Singleton {
  private static Singleton _instance;
  private Singleton(){}
  public static synchronized
      Singleton getInstance(){
    if(_instance == null)
      _instance = new Singleton();
    return _instance;
  }
}
Khi nào dùng & ưu điểmMột RandomNumber generator, một Restaurant/ShoppingCart, một KeyboardReader. Dễ đảm bảo key invariant; kiểm soát thời điểm tạo instance (quan trọng với object lớn). Ưu: lấy trách nhiệm quản lý instance khỏi lập trình viên (không thể tạo thêm); tiết kiệm bộ nhớ; tránh bug do nhiều instance. Comparator là singleton tốt vì không có state.
So với static method/classMột cách tránh tạo object là dùng static method (VD Math, System). Nhưng: Lacks flexibility — static method không truyền làm tham số/return được; Cannot be extended — static class không kế thừa/override được; Testability kém (khó mock, các test ảnh hưởng nhau); không viết extension method được. → Vì thế thường ưu tiên Singleton hơn static.

3 Template Method (Behavioral)

IntentĐịnh nghĩa khung xương của một thuật toán trong một operation, hoãn một số bước cho lớp con. Lớp con định nghĩa lại vài bước mà không thay đổi cấu trúc thuật toán. Nguyên tắc Hollywood: "Don't call us, we'll call you!"

Cấu trúc: AbstractClass có templateMethod() (gọi các step1..stepN), một số step là abstract/primitive; ConcreteClass hiện thực các primitive operation.

Ba chế độ của bước (steps)
  • Buộc lớp con hiện thực bước → abstract / interface.
  • Cho phép lớp con bổ sung bước (qua override) → hook non-abstract.
  • Cho phép lớp con định nghĩa lại bước → protected (overridable algorithm method).
  • Bước không cho override → final. Bước không kế thừa → private.
Beverage MachineTeaMaker / CoffeeMaker / CoffeeVerkeerdMaker có nhiều bước trùng (getCup, boilWater, pourInCup) & hành vi tùy chọn (addSugar/addLemon/addMilk) → gom vào BeverageMachine.makeBeverage() (template): boilWater() & pourInCup() là non-overridable; brew() abstract; addCondiments() override; addSugar() hook readonly.
Ưu điểm & ví dụ JavaTách cái thay đổi khỏi cái không đổi; phần code không đổi tái sử dụng dễ; lớp con chỉ lo khác biệt. Pattern này dùng mỗi khi extend một lớp trong Java: override paint() (hook) khi class MyFrame extends JFrame — thuật toán update() của JFrame gọi paint(). Applet lifecycle (init/start/stop/destroy/paint) cũng là Template Method. Ví dụ SocialNetwork: post() gọi logIn()/sendData()/logOut() abstract (Facebook, Twitter hiện thực khác nhau).

4 Factory Method (Creational)

Vấn đề: PizzaStore
Pizza orderPizza(String type){
  Pizza pizza;
  if(type.equals("cheese")) pizza = new CheesePizza();
  else if(type.equals("greek")) pizza = new GreekPizza();
  else if(type.equals("pepperoni")) pizza = new PepperoniPizza();
  pizza.prepare(); pizza.bake(); pizza.cut(); pizza.box();
  return pizza;
}
Trộn creation & preparation: đổi cách tạo pizza → vi phạm SRP; thêm/bớt loại pizza → vi phạm OCP.
Ý tưởngTách quá trình tạo (creation) khỏi sử dụng (usage): đóng gói code tạo object vào một lớp riêng (SimplePizzaFactory.createPizza(type)). Lớp đó vẫn phụ thuộc concrete class nhưng dependency đó không còn ảnh hưởng code preparation. (Simple Factory vẫn vi phạm OCP khi thêm loại → nâng lên Factory Method.)
Factory Method — StructureĐịnh nghĩa một interface để tạo object, nhưng để lớp con quyết định lớp nào được khởi tạo. Creator có factoryMethod() (abstract) & anOperation() dùng nó; ConcreteCreator override factoryMethod() trả về ConcreteProduct.
Khi nào dùng & cách hiện thựcDùng khi: client không cần/không được biết cách tạo object; thông tin tạo phức tạp/cồng kềnh → đơn giản hóa interface, khu trú logic khởi tạo; lớp cần lớp con chỉ định object tạo ra; không biết trước kiểu object cần tạo. Hiện thực: factory không nên bị instantiate (constructor private); dùng static method để tạo; interface đơn giản (đừng đòi nhiều tham số, có thể overload); thiết kế để tái sử dụng.
Ví dụ Java (factory có sẵn)DateFormat: getDateInstance(), getTimeInstance(), getDateInstance(FULL, Locale.FRANCE) — không truyền hàng loạt option, subtype tạo ra không cần biết. Toolkit: Toolkit.getDefaultToolkit() rồi tk.getImage("x.gif") (không new Image()). ImageFactory.loadImage() gói code cồng kềnh. BorderFactory.createRaisedBevelBorder(), KeyStroke.getKeyStroke() — tránh constructor quá nhiều "option/flag".

5 Abstract Factory (Creational)

Abstract FactoryKhi lớp factory cấp cao nhất & method tạo của nó đều abstract. Factory Method là factory có thể được khởi tạo & có method tạo overridable; Abstract Factory nâng lên mức "họ sản phẩm".
Khi nào dùngHệ thống nên độc lập với cách sản phẩm được tạo/kết hợp/biểu diễn; cấu hình được với một trong nhiều họ sản phẩm (families); một họ object liên quan phải dùng cùng nhau & cần cưỡng chế ràng buộc đó; muốn cung cấp thư viện sản phẩm chỉ lộ interface, không lộ cài đặt.
Ví dụ WidgetWidgetFactory (abstract) với CreateScrollBar(), CreateWindow(); MotifWidgetFactory & PMWidgetFactory tạo MotifWindow/PMWindow, MotifScrollBar/PMScrollBar. Client chỉ dùng interface trừu tượng.
Ưu (Good)Nhược (Bad)
Cô lập concrete class (client thao tác qua interface trừu tượng)Hỗ trợ kiểu sản phẩm MỚI khó
Đổi họ sản phẩm dễ (chỉ đổi ConcreteFactory)Phải sửa lại Abstract Factory & mọi lớp con
Cưỡng chế tính nhất quán giữa các sản phẩm(Nhưng không quá tệ trong ngôn ngữ định kiểu động)

1 Tổng quan Architectural Design

Mục đích"Đưa ra thiết kế có thể cài đặt và kiểm chứng được với yêu cầu". Kiến trúc được thiết kế từ yêu cầu phần mềm.

Đầu vào: Use-Case Model, Glossary, Supplementary Specifications, Software Architecture Document, Project Specific Guidelines. Đầu ra: Analysis Classes, Analysis Model, Use-Case Realization.

3 nội dung top-level: cấu trúc phần mềm & component · giao diện ngoài & giữa component · thiết kế database.

Chuỗi tinh chỉnhUse Cases → Analysis Classes → Design Elements → Source Code → Executable. Analysis classes là "bước đầu tiên hướng tới bản thực thi".

Nhắc lại lớp (Class): một trừu tượng mô tả nhóm object có chung Properties (attribute), Behavior (operation), Relationships, Semantics.

2 Ba loại Analysis Class

Toàn bộ hành vi của một use case phải được phân phối cho các analysis class. Dùng stereotype làm kim chỉ nam.

LoạiVai tròĐặc điểmKý hiệu
«boundary»Trung gian giữa giao diện & thứ bên ngoài hệ thốngPhụ thuộc môi trường (environment dependent)Vòng tròn có gạch ⊢○
«entity»Lưu & quản lý thông tin trong hệ thống (key abstraction)Độc lập môi trường (environment independent)Vòng tròn có gạch dưới

2.1 Boundary Classes

3 kiểu: User interface, System interface, Device interface. Quy tắc: một boundary class cho mỗi cặp actor/use-case. Mô hình hóa tương tác giữa hệ thống & môi trường.

GuidelineUI class: tập trung thông tin nào trình bày cho user, KHÔNG tập trung chi tiết UI. System/Device interface: tập trung giao thức nào cần định nghĩa, KHÔNG tập trung cách cài đặt. → "Tập trung vào trách nhiệm, không phải chi tiết!"

2.2 Entity Classes

Là key abstraction của hệ thống. Đầu vào: flow of events của use case, Business-Domain Model, Glossary.

Cách tìm (filtering nouns)Gạch chân cụm danh từ trong flow of events → loại: ứng viên trùng lặp · mơ hồ · là actor (ngoài phạm vi) · cấu trúc cài đặt · attribute (để sau) · operation.

2.3 Control Classes

Cung cấp hành vi điều phối; mô hình hành vi điều khiển đặc thù cho ≥1 use case. Phụ thuộc use case, độc lập môi trường.

GuidelineThường 1 control class cho mỗi use case. Use case đơn giản (chỉ thao tác dữ liệu lưu trữ) có thể bỏ control, chỉ dùng entity + boundary. Use case phức tạp cần ≥1 control (VD: transaction manager, resource coordinator, error handler).
Mẫu tương tác chuẩnActor ↔ boundary ↔ control ↔ entity. Boundary giao tiếp actor; control điều phối; entity giữ & quản lý thông tin.

3 Phân phối hành vi use case → lớp

Với mỗi flow of events của use case: (1) xác định analysis classes · (2) phân bổ trách nhiệm cho lớp · (3) mô hình tương tác bằng Interaction Diagram. Kết quả là một Use-Case Realization (gồm Sequence, Communication, Class diagrams).

Loại hành viGiao cho lớp
Giao tiếp với actorBoundary class
Dữ liệu đóng gói trong abstractionEntity class
Đặc thù cho use case / một luồng quan trọngControl class
Ai giữ trách nhiệm cần dữ liệu?Nếu một lớp có dữ liệu → đặt trách nhiệm ở đó. Nếu nhiều lớp có: đặt ở một lớp + thêm quan hệ; hoặc tạo lớp mới; hoặc đặt ở control class + thêm quan hệ tới các lớp cần thiết.

4 Interaction Diagrams

Thuật ngữ chung cho các sơ đồ nhấn mạnh tương tác giữa object: Sequence Diagram, Communication Diagram (+ biến thể Timing, Interaction Overview).

Sequence DiagramNhấn mạnh thứ tự thời gian của message. Thể hiện: object tham gia · trình tự message trao đổi. Thành phần: object lifeline, message, execution occurrence, reflexive message, đánh số phân cấp, interaction occurrence (ref).
Communication DiagramNhấn mạnh tổ chức (structural) của các object tham gia. Thể hiện: object · link giữa object · message trao đổi (có đánh số).

Sequence diagramCommunication diagram
Thể hiện trình tự message tường minhThể hiện quan hệ bên cạnh tương tác

Biến thểTiming diagram: ràng buộc thời gian của message. Interaction Overview: cái nhìn cấp cao ghép các tập tương tác thành chuỗi logic. Một Interaction diagram có thể không đủ → cần nhiều sơ đồ cho basic + các alternate flow.

5 Analysis Class Diagram

Từ Interaction diagram → rút ra trách nhiệm (responsibility) & quan hệ cho lớp.

  • Describe Responsibilities: mỗi message PerformResponsibility gửi tới :Supplier → thêm trách nhiệm vào lớp Supplier.
  • Finding Relationships: mỗi link trong communication diagram → một association trong class diagram ("relationship for every link").
  • Unify Analysis Classes: hợp nhất các lớp trùng vai trò để có class diagram gọn.

6 Architectural Patterns

Architectural patternGiải pháp tổng quát, tái sử dụng cho một vấn đề thường gặp trong kiến trúc phần mềm ở một ngữ cảnh (VD: performance, high availability). Nhiều pattern đã được cài trong framework: 3-layer/3-tier (DotNetNuke), MVC (Laravel, ASP.NET MVC), Microservices (Spring Boot, vert.x).

6.1 MVC (Model-View-Controller)

Classic (active) MVC

Model ↔ View ↔ Controller vòng tròn: Controller writes Model, Model fills View, Model notifies View (observer), View updates. Vấn đề? View phụ thuộc trực tiếp Model.

Passive MVC

Request → Routing → Controller → tương tác Model (data) → Controller gọi View → render. View không tự lắng nghe Model.

6.2 MVC vs 3-tier Architecture

3-tierMVC
Presentation Layer (.aspx, code-behind)View
Business Logic LayerController (+ logic)
Data Access Layer → DatabaseModel
Xếp tầng tuyến tính: tầng trên chỉ gọi tầng ngay dướiTam giác: 3 phần tương tác lẫn nhau
Front-end vs Back-endFront-end = UI & API, metadata, caching (phía user). Back-end = lưu trữ lớn, xử lý dữ liệu, phân tích chạy lâu. Hai bên có thể dùng các mẫu kiến trúc khác nhau.

6.3 Các mẫu khác

MVVM

View ↔ ViewModel ↔ Model. Data binding & commands giữa View–ViewModel; ViewModel cập nhật Model. Tách Presentation Logic khỏi Business Logic.

MVP (Android)

View (Activity/Fragment) ↔ Presenter ↔ Model (Interactor → REST/DB). Presenter làm trung gian.

Flux (React)

Luồng một chiều: Action → Dispatcher → Store → View → (Action…). Action Creators tạo action.

VIPER (iOS)

View · Interactor · Presenter · Entity · Routing (Wireframe). Tách trách nhiệm rất mịn.

Quản lý phiên bản (Version Control) & Git

Vì sao cần VCS, ba mô hình quản lý phiên bản, so sánh tập trung vs phân tán, từ vựng chuẩn và bộ lệnh Git cơ bản.

1 Vì sao cần Version Control?

Kịch bản cá nhânChương trình đang chạy → sửa "chỉ một thứ" → hỏng → sửa lại về cũ → vẫn hỏng — tại sao? Hoặc: hôm qua chạy tốt, tối cải tiến nhiều nhưng chưa xong, giờ phải nộp ngay.
Kịch bản nhómBạn sửa phần A (chạy), đồng nghiệp sửa phần B (chạy), ghép lại → không chạy. Một thay đổi ở phần này đã phá phần kia. Đâu là tất cả các thay đổi? Làm sao merge?
VCS làm gì
  • Giữ nhiều phiên bản (cũ & mới) của mọi thứ (không chỉ mã nguồn).
  • Yêu cầu comment cho mỗi thay đổi.
  • Cho check-in / check-out để biết ai đang làm file nào.
  • Hiển thị khác biệt giữa các phiên bản.
Công cụ diffdiff, rcsdiff, jDiff… hữu ích để tìm số ít khác biệt trong ít file. jDiff là plugin của jEdit: tô màu, cuộn đồng bộ, sửa trực tiếp — nhưng chỉ là công cụ diff, không phải giải pháp hoàn chỉnh.

2 Ba mô hình quản lý phiên bản

2.1 Lock-Modify-Unlock (khóa)

Chỉ một người được sửa file tại một thời điểm: check-out kèm khóa → sửa → commit & mở khóa. Người khác phải chờ.

Nhược điểmTuần tự hóa công việc, người khác phải đợi dù sửa vùng khác nhau của file → chậm, dễ tắc nghẽn.

2.2 Copy-Modify-Merge (sao chép–sửa–trộn)

Nhiều người cùng check-out (không khóa) & sửa bản sao cục bộ đồng thời. Ai commit trước thì xong; người sau khi commit gặp conflict → phải update + merge cục bộ rồi commit bản đã trộn.

Ưu điểmSong song hóa; đa số merge tự động được. Đây là mô hình phổ biến (SVN, Git…).

2.3 Distributed Version Control (phân tán)

Mỗi người clone toàn bộ repo về máy (có local repository riêng). Làm việc & commit cục bộ, sau đó push lên remote; nếu remote đã đổi → pull (fetch + merge) giải quyết conflict cục bộ rồi push lại.

4 Git — lệnh cơ bản

Git: hệ phân tán, làm việc với repo local & remote, có Git bash (CLI), miễn phí & mã nguồn mở.

# Khởi tạo / lấy repo
git init                       # tạo repo local ở thư mục hiện tại
git clone [remote-url]         # nhân bản repo remote về local
git remote add [name] [url]    # gán tên ngắn cho remote

# Vòng làm việc hằng ngày
git status                     # xem thay đổi cục bộ
git add [file]                 # chọn file cho commit (git add . = tất cả)
git commit -m "message"        # commit vào repo local
git pull                       # fetch + merge thay đổi mới nhất từ remote
git push [remote] [branch]     # đẩy thay đổi lên remote
Ví dụ luồng đầy đủ 
mkdir work && cd work
git clone https://github.com/SoftUni/test.git dir
cd test
# ...chỉnh sửa file...
git status
git add .
git commit -m "changes"
git push
Nơi host dự ánGitHub (#1, free cho open-source, có bản trả phí private), Google Code (SVN), SourceForge, CodePlex (TFS), Bitbucket (Mercurial, có private). Cài đặt: Windows dùng msysGit; Linux sudo apt-get install git.

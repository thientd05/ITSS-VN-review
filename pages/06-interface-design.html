1 Graphical User Interface Design

4 bước: (1.1) chuẩn hóa cấu hình màn hình · (1.2) tạo ảnh màn hình · (1.3) tạo sơ đồ chuyển màn · (1.4) tạo đặc tả màn hình.

1.1 Chuẩn hóa cấu hình màn hình

NhómChuẩn hóa gì
DisplayKích thước vật lý, độ phân giải, số màu hỗ trợ.
Screen (Window)Vị trí nút chuẩn (OK/Cancel/Register/Search), vị trí hiển thị message, tiêu đề & menu, nhất quán ký tự chữ-số, cách diễn đạt câu & mục chi tiết, phối màu.
ControlStyle/size/màu/ký tự; xử lý kiểm tra nhập (input check); trình tự di chuyển focus (tab sequence).
MenuThiết kế theo chuẩn common client area.
Direct keyboard inputNhất quán gán phím tắt (shortcut).
MessagesCách hiển thị khi tiến trình tốn thời gian (busy).
ErrorThực thi xử lý chuẩn hóa khi có lỗi.
HelpHelp chi tiết theo manual, nhất quán thuật ngữ & cách diễn giải.

1.2 Tạo ảnh màn hình (screen images)

Dựa trên use case & boundary class tương tác với user → ánh xạ boundary class thành màn hình; dựa mô tả input/output trong use-case spec/scenario → thiết kế màn hình bằng công cụ.

Công cụ GUIĐơn giản: Notepad, Office. Chuyên nghiệp free: InVision, Eclipse/NetBeans. Thương mại: Dreamweaver, Axure RP, Photoshop, Visual Studio.

1.3 Sơ đồ chuyển màn (Screen Transition Diagram) — 4 mẫu

Mẫu chuyểnÝ nghĩa
1. Simple screen transitionChuyển thông thường sang màn hình độc lập.
2. Transition to a dependent child screenMở pop-up con; khi con hiện, màn cha KHÔNG thao tác được.
3. Transition to an independent child screenMở pop-up con; cha & màn khác VẪN thao tác được khi con hiện.
4. Transition to a dependent screenMở màn phụ thuộc mới, kèm dữ liệu (selected data).

1.4 Đặc tả màn hình & field attributes

  • Screen image: ảnh màn hình sẽ hiển thị (đính bản in nếu có).
  • List of functions: tên các bộ phận (nút…), mô tả sự kiện, đặc tả input check & output.
  • Field attributes: quyết định thuộc tính của mục input/output; mỗi field gồm thuộc tính 1-byte (1 ký tự) ở đầu + mục biến đổi (VD: số ký tự, kiểu Numeral/Character, màu, canh lề, "error items blink").

2 System / Device Interface Design

3 bước: (2.1) Identify subsystem · (2.2) Identify subsystem interfaces · (2.3) Subsystem design.

2.1 Subsystem & Interface

SubsystemMột phần tử hiện thực (realize) một hoặc nhiều interface định nghĩa hành vi của nó. Đặc điểm: đóng gói hoàn toàn hành vi; đại diện một năng lực độc lập với interface rõ ràng (tiềm năng tái sử dụng); có thể mô hình nhiều biến thể cài đặt.
Package vs Subsystem
Subsystem
  • Cung cấp hành vi
  • Đóng gói hoàn toàn nội dung
  • Dễ thay thế (replaceable)
Package
  • KHÔNG cung cấp hành vi
  • KHÔNG đóng gói hoàn toàn nội dung
  • Có thể không dễ thay thế
Encapsulation là chìa khóa!

Subsystem raise the level of abstraction. Dùng để phân hoạch hệ thống thành phần: order/config/deliver/deploy độc lập; đổi mà không phá phần khác; giới hạn truy cập tài nguyên; đại diện sản phẩm/hệ thống ngoài.

Candidate subsystemsLớp cung cấp dịch vụ/tiện ích phức tạp; boundary class (UI & giao diện hệ thống ngoài); sản phẩm/hệ thống ngoài sẵn có (communication software, DB access, data structure, common utilities, application-specific products).

2.2 Identify Subsystem Interfaces

Mục đích: xác định interface của subsystem dựa trên trách nhiệm. Các bước: tìm tập interface ứng viên → tìm điểm tương đồng → định nghĩa dependency giữa interface → map interface tới subsystem → đặc tả hành vi interface → đóng gói interface.

Interface GuidelinesTên phản ánh vai trò (bắt đầu bằng "I", VD ICourseCatalogSystem). Description truyền tải trách nhiệm. Operation: tên phản ánh kết quả, mô tả rõ tham số & kết quả. Documentation: đính sequence/state diagram, test plan. → "Interface ổn định, well-defined là chìa khóa cho kiến trúc bền vững."
Modeling conventionInterface bắt đầu bằng "I". Required interface (đường có ổ cắm ○–) vs Provided interface (đường có bóng ─○). VD: RegistrationController cần ICourseCatalogSystem (required); subsystem CourseCatalogSystem cung cấp (provided).

2.3 Subsystem Design

Các bước: (1) phân phối hành vi subsystem cho phần tử · (2) tài liệu hóa phần tử · (3) mô tả dependency của subsystem · (4) checkpoints.

Subsystem Design — Goals & SuggestionsGoals: Loose coupling · Portability/plug-and-play · Insulation from change · Independent evolution. Strong suggestions: KHÔNG lộ chi tiết — chỉ interface; chỉ phụ thuộc vào interface khác. → "Chìa khóa là abstraction & encapsulation."
GatesMột điểm kết nối trong tương tác cho message đi vào/ra khỏi tương tác — điểm trên biên của sequence diagram; tên message kết nối = tên gate. Phân biệt Black-box view (subsystem interaction) và White-box view (Internal structure: Subsystem Manager điều phối các Design Element bên trong).
Subsystem DependenciesƯu tiên phụ thuộc vào subsystem qua interface (flexible, preferred); phụ thuộc vào package thì dùng cẩn thận (use with care).
Checkpoints: Design SubsystemsCó realization association cho mỗi interface được cung cấp? Có dependency cho mỗi interface được dùng? Không phần tử nào trong subsystem có public visibility? Mỗi operation của interface có được tài liệu hóa trong interaction diagram (1:1 mapping)?

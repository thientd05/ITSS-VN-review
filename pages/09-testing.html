1 Tổng quan Testing & QA

Software testThực thi chương trình để xác định một tính chất (property) của nó có đúng hay không. Test pass nếu property holds, fail nếu không holds trong lần chạy đó.

QA (Software Quality Assurance) = testing + nhiều hoạt động khác: static analysis (đánh giá không chạy code), proofs of correctness, code review, software process… "Không hoạt động đơn lẻ nào đảm bảo được chất lượng phần mềm."

2 Black box vs White box

Black boxWhite box
Chọn inputKhông biết cài đặt (dựa spec/hành vi)Có biết cài đặt
Tập trungHành vi của SUT; cần "soict" (kết quả kỳ vọng)Coverage (độ phủ)
Ý tưởngSUT/Implementation phải khớp spec (pre/postcondition)Nếu test suite không chạy qua một statement → statement đó có thể lỗi

Quy trình chung (A→B→C→D): chọn input → định nghĩa expected outcome (soict) → chạy SUT → so sánh kết quả với soict. "It's not black-and-white, but…" — thực tế phối hợp cả hai.

3 Kỹ thuật Black-box

3.1 Equivalence Partitioning (phân vùng tương đương)

Chia không gian input thành các lớp tương đương; mọi input trong cùng lớp cho cùng kết quả → mỗi lớp lấy 1 test case.

Examination Judgment ProgramPassed nếu: cả Math & Physics ≥ 70; hoặc trung bình Math & Physics ≥ 80. Ngược lại Failed. → 7 lớp tương đương ⇒ ≥7 test case. Quan trọng: phải thêm test cho "invalid value" (VD Math=-15, Physics=120…) — chỉ hợp lệ là chưa đủ.

3.2 Boundary-value Analysis (phân tích giá trị biên)

Lấy test data ở biên → nhiều defect nằm ở đó (VD lập trình viên viết if(mathscore > 70) thay vì >= 70).

Biên của điểm Math (ngưỡng 70)Invalid value | -1 0 … failed … 69 70 passed … 100 101 | Invalid value. (Kiểm cả 69/70 & 79/80 cho trung bình.)

3.3 Decision Table (bảng quyết định)

Biểu diễn quan hệ giữa điều kiện & nội dung xử lý dưới dạng bảng — công cụ tốt khi nhiều điều kiện kết hợp.

Ví dụ: report từ file nhân viênĐiều kiện: Under age 30 (Y/N) · Male (Y/N) · Married (Y/N) → chọn Output Report 1–4 (đánh dấu X theo tổ hợp). Với Examination Judgment: Condition1 Math≥70, Condition2 Physics≥70, Condition3 avg≥80 → các cột TC cho "Passed"/"Failed".

3.4 Testing for Use Case

Từ use case → test cases: xác định mọi scenario; luồng thay thế vẽ thành graph theo từng action; tạo test cho basic flow + mỗi alternative flow + tổ hợp hợp lý + vòng lặp vô hạn.

Test cases "Log in""Thành công" (PIN đúng) · "Thất bại" (PIN sai & số lần sai < 3) · "Khóa tài khoản" (PIN sai & số lần sai = 3). Kết hợp boundary: số lần sai = 2, 4.

4 Kỹ thuật White-box

Test case phải phủ mọi cấu trúc xử lý trong code.

Độ phủCông thứcÝ nghĩa 100%
C0 (statement)số statement đã chạy / tổng statementMọi câu lệnh được thực thi
C1 (branch)số nhánh đã đi / tổng nhánhMọi nhánh được thực thi
Giới hạn White-boxNgăn statement/branch bị bỏ sót, nhưng KHÔNG phát hiện được chức năng chưa được cài đặt (vì không có code thì không có gì để phủ).
Control flow test100% C0 có thể chỉ cần 2 test (TC1int, TC2int) nhưng chưa phủ hết nhánh. 100% C1 cần thêm test (TC3int…). C1 mạnh hơn C0 nhưng số test vẫn có thể ít hơn số lớp tương đương của black-box (vì black-box còn xét invalid/functional). → Kết hợp cả hai.

Testing vòng lặp (loop)

Thay C0/C1, dùng 6 ca (dựa boundary-value): Skip vòng lặp · chạy 1 lần · m lần điển hình · n-1 · n · n+1 (n = số tối đa, m = số điển hình, m<n).

n=50số SV = 0, 1, 20 (điển hình), 49, 50, 51(→invalid). Nested loop: cố định vòng ngoài ở số điển hình, test vòng trong như loop đơn; rồi đổi vai.
Data/message path test (integration)Dùng sequence chart để white-box test tích hợp: chạy mọi message path/flow → 100% message path coverage.

5 Kết hợp Black/White & quy trình test hiệu quả

Ưu Black boxTest bao trùm dựa external spec; nền tảng & mạnh để tạo phần mềm chất lượng cao.
Ưu White boxNếu path/flow không xuất hiện trong spec đã viết → black-box có thể bỏ sót; white-box bắt được (VD if 0 =< score =< 100).
Quy trình test hiệu quả & đầy đủ(1) Test theo external spec — nếu pass hết ⇒ mọi external spec cài đúng. (2) Thêm test theo internal spec (path/flow còn lại) — nếu pass với coverage 100% ⇒ mọi chức năng cài đúng, không có mã dư thừa.

6 JUnit

Công cụ cho test-driven development (junit.org), tích hợp nhiều IDE (Eclipse, BlueJ…). Họ xUnit có bản cho nhiều ngôn ngữ.

Viết TestCase (kiểu cổ điển)Kế thừa junit.framework.TestCase; tên lớp dạng TestMyClass/MyClassTest; method test dạng testXxx() — theo programming by contract: set up precondition → thực thi chức năng → kiểm postcondition.
Cơ chếVai trò
setUp() / tearDown()Tạo/giải phóng test fixture trước/sau mỗi test → mỗi test bắt đầu với đối tượng "mới tinh", không phụ thuộc thứ tự chạy.
Assert methodsassertTrue/assertFalse, assertNull/assertNotNull, assertEquals (dùng equals), assertSame/assertNotSame (dùng ==). Tham số: message, expected, actual; nên kèm message.
TestRunnerChạy test & báo cáo TestResult (Failure vs Error).
Kiểu JUnit 4 (annotation) 
import org.junit.*;
import static org.junit.Assert.*;
public class RandomHelloTest {
  @Test
  public void test_NoException(){
    RandomHello.getGreeting();
    assertTrue(true);          // chạy & return, không ném exception
  }
  @Test(expected = ArrayIndexOutOfBoundsException.class)
  public void testBadIndex(){
    new ArrayIntList().get(4);  // mong đợi exception → pass
  }
}
Failure vs Error & hintsFailure: test không pass (oracle sai). Error: có sự cố ngoài dự kiến (VD null pointer). Green bar = pass, red bar = fail/error. Hints: test một thứ mỗi method (10 test nhỏ > 1 test lớn); tiết kiệm assert (assert đầu fail sẽ dừng); test độc lập thứ tự; dùng @Test(expected=...) thay try/catch; đặt tên test mô tả (thường rất dài); @RunWith(Suite.class) gom test suite.
RandomHello — kiểm tính ngẫu nhiênTest getGreeting: (1) chạy không ném exception; (2) trả về một trong các greeting đã định nghĩa; (3) chạy 100 lần, nếu có greeting không bao giờ xuất hiện → khó là ngẫu nhiên; (4) dùng thống kê chi-squared để bắt "sleazy developer" chỉ luân phiên 2 greeting.

7 Integration Test

Kiểm giao diện giữa các module cùng input/output.

Stub / DriverChương trình giả lập chức năng của module cấp thấp hơn (stub) hoặc cấp cao hơn (driver).
Cách tiếp cậnĐặc điểm
Top-downPhát hiện sớm defect do hiểu sai spec; tốt cho hệ thống mới; cần stub (đơn giản, trả một giá trị).
Bottom-upModule cấp thấp độc lập → test song song; tốt khi sửa hệ thống có sẵn; cần driver (phức tạp hơn, có điều khiển).
Big-bangGhép tất cả module đã unit-test cùng lúc; giảm số thủ tục test ở chương trình nhỏ; nhưng khó định vị lỗi.
SandwichModule cấp thấp test bottom-up, cấp cao test top-down.
Regression Test"Khi sửa một bug, bạn tạo ra vài bug mới." → Test lại sau khi code đổi để chắc vẫn chạy đúng: chạy lại test case cũ; kiểm side-effect. Chạy càng thường xuyên càng tốt, dùng công cụ regression tự động.

1 Mục đích & vị trí trong lifecycle

Mục đíchPhân tích tương tác của analysis class → xác định các phần tử của Design Model (Design classes, Package, Subsystem). Thực hiện ở bước "Refine the Architecture" (do Architect).

Đầu vào: Analysis Model, Software Architecture Document, Supplementary Specifications, Project Specific Guidelines → Đầu ra: Design Model (design elements).

3 Package — nhóm các Design Class

ClassMô tả tập object chia sẻ cùng responsibilities, relationships, operations, attributes, semantics.
PackageCơ chế tổng quát để tổ chức các phần tử thành nhóm; là một model element có thể chứa các model element khác. Dùng để: tổ chức mô hình đang phát triển · làm đơn vị quản lý cấu hình (configuration management).

Tiêu chí đóng gói (packaging criteria)

  • Configuration units (đơn vị cấu hình).
  • Phân bổ nguồn lực giữa các nhóm phát triển.
  • Phản ánh loại user.
  • Đại diện sản phẩm/dịch vụ sẵn có mà hệ thống dùng.

4 Packaging Tips

4.1 Boundary Classes

Nếu giao diện DỄ thay đổi nhiềuĐặt boundary class vào package riêng (cô lập thay đổi).
Nếu KHÓ thay đổiĐóng gói boundary cùng các lớp liên quan chức năng.

4.2 Functionally Related Classes — dấu hiệu "liên quan chức năng"

  • Thay đổi hành vi/cấu trúc lớp này bắt buộc đổi lớp kia.
  • Xóa lớp này ảnh hưởng lớp kia.
  • Hai object trao đổi nhiều message hoặc giao tiếp phức tạp.
  • Boundary liên quan chức năng tới một entity nếu nhiệm vụ của nó là trình bày entity đó.
  • Hai lớp tương tác với / bị ảnh hưởng bởi cùng một actor.
  • Có quan hệ với nhau; một lớp tạo instance của lớp kia.
KHÔNG đặt chung package khiHai lớp liên quan tới các actor khác nhau; hoặc một lớp optional và một lớp mandatory.

4.3 Package Element Visibility

EncapsulationChỉ các lớp public mới được tham chiếu từ ngoài package sở hữu. Lớp private (kí hiệu -) bị ẩn. → Nguyên lý OO: Encapsulation.

5 Package Coupling — quy tắc phụ thuộc

3 vi phạm coupling (đánh dấu X)
  • Package không được cross-coupled (phụ thuộc vòng lẫn nhau).
  • Package ở tầng dưới không được phụ thuộc package ở tầng trên.
  • Nhìn chung, phụ thuộc không nên nhảy tầng (skip layers).
Ví dụ Registration PackageMainStudentForm ◇ RegisterForCoursesForm «boundary» → RegistrationController «control». MainRegistrarForm ◇ CloseRegistrationForm «boundary» → CloseRegistrationController «control». (Boundary phụ thuộc vào control cùng package.)
External System Interfaces PackageChứa các «Interface» như IBillingSystem, ICourseCatalogSystem — cô lập hệ thống ngoài. University Artifacts Package chứa entity (Student, CourseOffering, Course, Professor, Schedule…) với generalization & association.

6 Subsystem (giới thiệu)

Ngoài package, một analysis class phức tạp có thể trở thành subsystem — đơn vị đóng gói hành vi sau một interface. Chi tiết ở chủ đề Thiết kế giao diện (Subsystem & Interface, Package vs Subsystem, gates).
